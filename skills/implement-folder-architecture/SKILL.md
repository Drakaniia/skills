# Implement Folder Architecture

Use this skill to act as a **Structural Refactoring Engineer** — systematically executing a folder architecture migration based on an existing audit report or user-defined architecture. This skill bridges the gap between **detecting** structural issues (the domain of `audit-codebase`) and **preventing** future decay (the domain of `folder-architecture`).

## Core Principles

- **Incremental safety**: Perform the migration one phase at a time, verifying the project builds and tests pass after each phase.
- **Preserve functionality**: All public APIs, exports, and imports must work identically before and after the migration.
- **Follow existing conventions**: Match the project's detected naming convention and architecture pattern — do not introduce new conventions during a migration.
- **Language-agnostic**: Every operation applies to any language. Language-specific mechanics are in the shared reference files.
- **Report-driven**: When an audit report exists, follow it strictly. The report is the implementation contract.
- **Test before, verify after**: Write or identify tests that capture the current behavior before moving code, then verify after each phase.

## Workflow

```
audit-codebase
        │
        ▼
Generates implementation plan (report)
        │
        ▼
implement-folder-architecture
        │
        ├── Phase 1: Quick Cleanup (safe, low-risk)
        ├── Phase 2: File Refactoring (medium effort)
        └── Phase 3: Structural Refactoring (high effort)
        │
        ▼
Verify build + tests pass
        │
        ▼
folder-architecture (prevents future decay)
```

## When This Skill Activates

Use this skill when:

- An `audit-codebase` report has been generated and needs to be executed
- A user explicitly asks to reorganize the project structure
- A systematic folder refactoring is needed (splitting bloated directories, flattening deep nesting)
- Moving files between directories in bulk and need import paths updated
- Implementing a predefined folder architecture (e.g., switching from layered to feature-first)

## Pre-Migration Preparation

Before executing any changes, gather the following context:

### 1. Read the Audit Report (if available)

- Read the `<repo-name>-audit.md` file generated by `audit-codebase`
- Identify the three phases defined in the report
- Note the before/after diagrams, proposed file locations, and import changes

### 2. Understand the Project

- **Language & framework**: Identify the primary language and framework from `package.json`, `Cargo.toml`, `requirements.txt`, `go.mod`, etc.
- **Naming convention**: Scan 5-10 file names in `src/` or the equivalent source directory to detect kebab-case, snake_case, camelCase, or PascalCase
- **Architecture pattern**: Is it feature-first, layered, or hybrid? Check 2-3 directories for clues
- **Barrel files**: Check for `index.ts`, `__init__.py`, `mod.rs` files to understand the module export pattern
- **Build system**: How does the project build? (`npm run build`, `cargo build`, `go build ./...`, `mvn compile`, etc.)
- **Test runner**: How are tests run? (`npm test`, `pytest`, `go test ./...`, `cargo test`, etc.)

### 3. Snapshot the Current State

- Record the current directory tree of affected areas (or take note of the before-diagrams in the report)
- Note any existing tests that cover the code being moved
- Verify the current state builds and tests pass before any changes

## Phase Execution

Execute the migration in three phases, exactly as ordered in the audit report. **Do not skip phases** — Phase 1 sets the foundation for Phase 2, which sets the foundation for Phase 3.

### Phase 1: Quick Cleanup

_Low effort, safe changes. No code logic is altered._

Operations:

- **Remove empty/dead directories** — delete directories that are truly empty
- **Consolidate doc files** — move `.md` files into a `docs/` directory
- **Rename files for naming consistency** — standardize casing within a directory
- **Remove compiled/binary artifacts** — delete files that shouldn't be in source control

**Procedure for each cleanup task:**

1. Verify the file/directory is safe to remove (check `.gitignore`, check for hidden references)
2. For renaming: use `git mv` or equivalent to preserve history
3. After renaming, use code-search to find and update any references to the old name
4. Commit after Phase 1, or at minimum verify the build still passes

**Commands:**

```bash
# Find empty directories
find . -type d -empty ! -path '*/node_modules/*' ! -path '*/.git/*'

# Rename a file (preserves git history)
git mv old-name.py new-name.py
```

**Barrel file updates for renames:**

If a renamed file was exported from a barrel file, update the barrel:

| Language               | Barrel update                                                  |
| ---------------------- | -------------------------------------------------------------- |
| Python (`__init__.py`) | `from .new_name import X` instead of `from .old_name import X` |
| JS/TS (`index.ts`)     | `export { X } from './new-name'` instead of `./old-name`       |
| Rust (`mod.rs`)        | `pub mod new_name;` instead of `pub mod old_name;`             |

**Verification after Phase 1:**

- [ ] Project builds successfully
- [ ] All tests still pass
- [ ] No lingering references to removed/renamed files

### Phase 2: File Refactoring

_Medium effort — splitting oversized files and moving misplaced files._

Operations:

- **Split oversized files** (>400 lines) into smaller modules by natural boundaries
- **Move misplaced files** to their correct locations
- **Update all import paths** across the codebase

#### Splitting Oversized Files

For each file exceeding the 400-line threshold:

**Step 1: Identify natural boundaries**

Read the file and identify modular sections. Look for:

- Comment headers dividing sections (`# ─── Validation ───`)
- Distinct class/function definitions that serve different concerns
- Import-group clusters (imports that only certain sections use)
- Test functions that test specific sections

**Step 2: Determine split structure**

| Condition             | Recommendation                             |
| --------------------- | ------------------------------------------ |
| 1-2 extracted modules | Same directory, flat files                 |
| 3+ extracted modules  | Subdirectory with barrel file              |
| Clear domain boundary | Subdirectory even with 2 files             |
| Cross-cutting utility | Extract to `utils/` or `shared/` directory |

**Step 3: Apply language-specific splitting mechanics**

Load the appropriate language reference:

- [SPLITTING-GUIDE.md](references/SPLITTING-GUIDE.md) — Comprehensive language-specific splitting mechanics with before/after code examples for Python, JS/TS, Go, Rust, Java, C#, Ruby, and PHP

See also the shared reference in `audit-codebase` or `folder-architecture` skills for the same canonical content.

**Step 4: Create new files and barrel**

1. Create the extracted module files with appropriate exports
2. Create or update barrel files to re-export public symbols
3. Update the original file to import from the new modules (trim to orchestration)

**Step 5: Update all consumers**

Use code-search tools to find every file that imports from the original module:

```bash
# Python
grep -r "from <original_module> import" --include="*.py" .

# JS/TS
grep -r "from './<original>'\|from '../<original>" --include="*.{ts,tsx,js,jsx}" .

# Go
grep -r "\"<module-path>/<package>\"" --include="*.go" .

# Rust
grep -r "use crate::.*::" --include="*.rs" .
```

Update each consumer's import path to match the new module structure. Use this table pattern:

| Old Import                           | New Import                              |
| ------------------------------------ | --------------------------------------- |
| `from order_service import validate` | `from order_validation import validate` |
| `from ..service import X`            | `from ..order.service import X`         |

**Step 6: Verify**

- [ ] Project compiles/builds
- [ ] Tests pass
- [ ] Original file is trimmed (not deleted if it still has orchestration code)
- [ ] Barrel files correctly re-export all public symbols
- [ ] No circular imports introduced

#### Moving Misplaced Files

For files that belong in a different directory:

**Step 1: Check for dependencies**

- Read the file being moved — note all its imports and exports
- Search for all files that import from this file

**Step 2: Move the file**

```bash
# Move (preserves git history)
git mv src/current-location/file.ext src/target-location/file.ext

# Or create target directory and move
mkdir -p src/target-location
git mv src/current-location/file.ext src/target-location/file.ext
```

**Step 3: Update import paths**

- Update imports in the moved file (relative paths may change)
- Update imports in all consumer files (import paths changed)
- Update barrel files that exported the moved symbol

**Step 4: Verify**

- [ ] Project builds
- [ ] Tests pass
- [ ] No file references the old import path

### Phase 3: Structural Refactoring

_Higher effort — reorganizing folders, flattening deep nesting, splitting bloated directories._

Operations:

- **Split bloated directories** (>30 files) into subdirectories by domain/concern
- **Flatten deep nesting** (>4 levels) by merging related directories
- **Architecture migration** — convert from layered to feature-first (or vice versa)

#### Splitting Bloated Directories

For each directory exceeding the 30-file threshold:

**Step 1: Review the proposed structure**

Check the audit report's proposed after-tree. If no report exists, create a grouping plan:

- Group files by domain/feature/concern
- Name subdirectories using the project's naming convention

**Step 2: Create subdirectories**

```bash
mkdir -p src/utils/string-helpers
mkdir -p src/utils/date-helpers
mkdir -p src/utils/network
```

**Step 3: Move files in batches**

Move files in small batches (3-5 files at a time), updating imports after each batch:

1. `git mv` the files into the new subdirectory
2. Update import paths in the moved files (if any)
3. Update import paths in all files that import from the moved files
4. Update barrel files
5. Verify build passes
6. Repeat

**Step 4: Create barrel files for new subdirectories**

For each new subdirectory that has 3+ public modules, create a barrel file:

| Language | Barrel file   | Example content                                           |
| -------- | ------------- | --------------------------------------------------------- |
| Python   | `__init__.py` | `from .validation import validate_order`                  |
| JS/TS    | `index.ts`    | `export { validateOrder } from './validation'`            |
| Rust     | `mod.rs`      | `pub mod validation; pub use validation::validate_order;` |

**Step 5: Update parent barrel**

If the parent directory had a barrel file, update it to either:

- Re-export from subdirectory barrels (if preserving thin facade)
- Remove old re-exports and let consumers import from subdirectories directly

#### Flattening Deep Nesting

For paths deeper than 4 levels:

**Step 1: Identify the target structure**

Plan a flatter tree by merging related directories:

```
Before:                              After:
app/modules/admin/settings/          app/modules/
└── permissions/                       ├── admin-settings/
    └── roles/                         └── admin-permissions/
```

**Step 2: Collapse one level at a time**

1. Identify two consecutive levels that can be merged
2. `git mv` files from child to parent
3. Update all import paths
4. Remove the now-empty child directory
5. Verify build passes
6. Repeat for remaining depth

**Step 3: Update imports** — all relative import paths that referenced the old nesting must be updated.

**Step 4: Update barrel files** — any `mod.rs`, `__init__.py`, or `index.ts` files affected by the flattening must be updated.

#### Architecture Migration (e.g., Layered → Feature-First)

This is the most complex operation. Follow these steps:

**Step 1: Map current structure to target structure**

Create a mapping table:

| Current location                     | Target location                         |
| ------------------------------------ | --------------------------------------- |
| `src/controllers/user_controller.py` | `src/features/users/user_controller.py` |
| `src/models/user.py`                 | `src/features/users/user_model.py`      |
| `src/services/user_service.py`       | `src/features/users/user_service.py`    |

**Step 2: Create target directory skeleton**

```bash
mkdir -p src/features/users
mkdir -p src/features/billing
mkdir -p src/features/notifications
```

**Step 3: Move one feature at a time**

For each feature:

1. `git mv` all files belonging to that feature into the new directory
2. Update all import paths (in moved files + consumers)
3. Create barrel file for the feature directory
4. Verify build passes

**Step 4: Remove old directories that are now empty**

**Step 5: Update shared code**

Code that remains cross-cutting (e.g., logging, database connection, shared types) stays in `shared/` or is moved to `src/shared/`.

## Import Refactoring Protocol

After any file move or rename, follow this exact protocol:

### Step 1: Find All Consumers

Search for all files that reference the moved file by its old path:

| Language | Search Pattern                                                                 |
| -------- | ------------------------------------------------------------------------------ |
| Python   | `grep -r "from .*<old_path>" --include="*.py"`                                 |
| JS/TS    | `grep -r "from ['\"]\\.\\.*/<old_path>" --include="*.{ts,tsx,js,jsx}"`         |
| Go       | `grep -r "\"<module>/<old_package>\"" --include="*.go"`                        |
| Rust     | `grep -r "use crate::<old_path>" --include="*.rs"`                             |
| Java     | `grep -r "import <old_package>" --include="*.java"`                            |
| C#       | `grep -r "using <old_namespace>" --include="*.cs"`                             |
| Ruby     | `grep -r "require.*<old_path>\|require_relative.*<old_path>" --include="*.rb"` |
| PHP      | `grep -r "use <old_namespace>" --include="*.php"`                              |

### Step 2: Update Imports

For each consumer, update the import path:

- **Relative imports**: Compute the new relative path from consumer to target
- **Absolute imports**: Update the absolute module/package path
- **Barrel imports**: If importing through a barrel, update barrel first

### Step 3: Verify No Orphaned References

After all updates, confirm no file in the project still references the old path:

```bash
grep -r "<old_path_keyword>" --include="*.py" --include="*.ts" --include="*.js" --include="*.go" --include="*.rs" --include="*.java" --include="*.cs" --include="*.rb" --include="*.php" .
```

## Barrel File Management

### When to Create Barrel Files

| Condition                                        | Action                             |
| ------------------------------------------------ | ---------------------------------- |
| Directory has 3+ public modules                  | Create barrel file                 |
| Splitting a file into 3+ modules in subdirectory | Create barrel file in subdirectory |
| Moving files from a directory with a barrel      | Update barrel to reflect new paths |
| Directory has ≤2 files                           | No barrel needed                   |

### Barrel File Content by Language

**Python** (`__init__.py`):

```python
from .module_a import PublicSymbolA
from .module_b import PublicSymbolB

__all__ = ["PublicSymbolA", "PublicSymbolB"]
```

**JavaScript/TypeScript** (`index.ts`):

```typescript
export { PublicSymbolA } from "./module-a";
export { PublicSymbolB } from "./module-b";
```

**Rust** (`mod.rs`):

```rust
pub mod module_a;
pub mod module_b;

pub use module_a::PublicSymbolA;
pub use module_b::PublicSymbolB;
```

**Go**: No barrel file needed — same-package visibility is automatic.

**Java/C#/Ruby/PHP**: Barrel files are not standard. Use import comments or documentation instead.

### Barrel File Update Checklist

After each structural change:

- [ ] Check if the moved/created directory already has a barrel file
- [ ] If yes, add/update re-exports for the affected symbols
- [ ] If no, check if 3+ public modules now exist in the directory
- [ ] Remove re-exports from barrel files for symbols that were moved out
- [ ] Verify consumers can still import through the barrel

## Safety Guidelines

### Incremental Migration

- **Move one thing at a time**: One file, one directory, one feature per step
- **Verify after each step**: Build check, test run — never skip
- **Commit frequently**: After each verified step, commit with descriptive message

### Commit Strategy

```text
Phase 1: Quick Cleanup
  feat: remove empty directories src/legacy/ and src/archive/
  feat: consolidate markdown files into docs/
  feat: standardize snake_case naming in src/

Phase 2: File Refactoring
  refactor: split order_service.py into 4 modules
  refactor: move database config to config/

Phase 3: Structural Refactoring
  refactor: split src/utils/ into domain subdirectories
  refactor: flatten app/modules/admin/settings/ to app/admin-settings/
  refactor: migrate src/ from layered to feature-first structure
```

### Rollback Plan

If a step fails verification:

1. **Stop** — do not proceed to the next step
2. **Diagnose** — identify the issue (broken import, missing export, circular dependency)
3. **Fix** — correct the issue (don't move forward, fix the current step)
4. **Re-verify** — build + tests must pass
5. If the issue cannot be fixed quickly, **revert** the last change:

```bash
git checkout -- <affected-files>
# or
git stash
```

### What NOT to Do

- ❌ Do not modify business logic while moving files — refactoring is structural only
- ❌ Do not combine multiple moves in one step — isolate each change
- ❌ Do not delete the original file until all consumers are updated
- ❌ Do not change naming conventions mid-migration — rename first (Phase 1), then restructure (Phase 3)
- ❌ Do not skip build verification — always verify after each step

## Verification Protocol

### Per-Step Verification

After each individual change (file move, rename, split):

1. **Build check**: Run the project's build command
   - `npm run build` / `tsc --noEmit` (JS/TS)
   - `python -c "import <package>"` (Python)
   - `go build ./...` (Go)
   - `cargo check` (Rust)
   - `mvn compile` / `gradle build` (Java)
   - `dotnet build` (C#)
   - `rails runner "puts 'ok'"` (Ruby)
   - `composer dump-autoload && php artisan optimize` (PHP)

2. **Test run**: Run the project's tests
   - `npm test` / `jest`
   - `pytest`
   - `go test ./...`
   - `cargo test`
   - `mvn test`
   - `dotnet test`
   - `rspec` / `rails test`
   - `phpunit` / `pest`

3. **Lint/type check** (if applicable):
   - `tsc --noEmit` (TypeScript)
   - `mypy` (Python)
   - `go vet` (Go)
   - `cargo clippy` (Rust)

### Phase Completion Verification

After completing all tasks in a phase:

- [ ] Project builds without errors
- [ ] All tests pass
- [ ] No lint/type errors
- [ ] No circular imports introduced
- [ ] All barrel files are up to date
- [ ] Public API surface is preserved (consumers unchanged)
- [ ] No source files remain in the project root
- [ ] No empty directories remain
- [ ] The audit report's progress checklist is updated

### Final Verification

After all three phases are complete:

- [ ] Full build passes
- [ ] Full test suite passes
- [ ] Type checker and linter pass
- [ ] No remaining references to old paths (code-search the entire project)
- [ ] The `folder-architecture` skill is now active to prevent future decay
- [ ] Commit all changes with a summary message

## Edge Cases

| Situation                                              | Handling                                                                                                                 |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------ |
| **No audit report available**                          | Use `ask_user` tool to ask the user to run `audit-codebase` first, or use `folder-architecture` thresholds to guide the refactoring manually    |
| **Empty target directory after migration**             | Remove it, or verify the directory serves a purpose (e.g., placeholder for future code)                                  |
| **Circular dependencies found during refactoring**     | Extract the shared dependency into a separate module and update both sides to import from it                             |
| **File imported by 20+ consumers (high blast radius)** | Keep the original file as a thin re-export facade, update consumers in batches of 5, verify after each batch             |
| **Mixed-content file (component + logic + styles)**    | Extract styles first, then sub-components, then logic/hooks — keep the component boundary clean                          |
| **File with no clear internal boundaries**             | Flag for manual review, do not attempt to split — move the file as-is if needed, but document the need for decomposition |
| **Generated code**                                     | Skip migration for `generated/`, `dist/`, `build/` directories — they follow their own rules                             |
| **Vendor/third-party code**                            | Skip `node_modules/`, `vendor/`, `.venv/` — do not restructure external dependencies                                     |
| **Monorepo with packages**                             | Apply migration per-package, not globally. Each package has its own structure and thresholds                             |
| **Build fails after a step**                           | Revert the change, diagnose the issue, fix, and retry. Never skip build verification                                     |
| **User disagrees with proposed structure**             | Respect user's preference. Move files where they specify, but flag any structural concerns                               |
